# 架构

一个自托管的 AI 陪伴后端：多入口（Telegram + 网页 PWA）共享**同一份记忆、同一个人**，由 Claude Code 驱动，长期在线、有主动性、可接语音/推送。

本文讲清楚整套东西怎么搭起来、为什么这么搭。具体接口见 [`API.md`](API.md)。

---

## 一句话核心

> **不是让多个入口互相「同步」记忆，而是从头到尾只有一份记忆。**

TG 和网页只是两个「壳」，背后打的是同一个网关（hub）、读写同一份数据。网上多数教程要靠 GitHub / 云盘中转，是因为两端跑在不同机器上；只要把所有入口收拢到同一个后端，「同步」这个问题就不存在了。加第三个入口（邮件、语音…）时，它一接上网关就继承了全部记忆，客户端零改动。

## 组件全景

```
   Telegram Bot(s)          网页 PWA (React)
   聊天入口 ①                聊天入口 ②
        │                        │
        └──────────┬─────────────┘
                   ▼
        ┌───────────────────────────────────┐
        │  HUB (Bun / TypeScript, 单文件)     │   ← 网关：一切都经过它
        │  · HTTP + 客户端 WS (:3456)         │
        │  · 每次回复前注入上下文             │
        │  · 解析 AI 回复里的「标记」并执行   │
        │  · 主动性引擎(定时 tick)            │
        └───────┬───────────────────┬────────┘
                │ bridge WS(:3457)  │ spawn `claude -p`
                │ (127.0.0.1)       │ (一次性任务:做梦/日记/情绪…)
                ▼                   ▼
        ┌──────────────┐    ┌──────────────────┐
        │ Claude Code  │    │ 一次性 cc 进程    │
        │ 持久会话(cc) │    │ (独立会话,不污染) │
        │ = 主 AI 的"人"│    └──────────────────┘
        └──────┬───────┘
               │ MCP 工具(reply / 记忆读写)
               ▼
        ┌──────────────────────────────────────┐
        │ 共享存储:memory.db(SQLite) + *.json    │
        │ 记忆 · 两入口的聊天记录 · 日记 · 任务  │
        │ · 情绪 · 账本 · 收藏 …                 │
        └──────────────────────────────────────┘
```

### hub（网关）
单文件 Bun/TypeScript 服务，是整个系统的心脏。职责：
- 对外：HTTP API + 客户端 WebSocket（前端用），Telegram long-polling（一或多个 bot）。
- 对内：通过一条只监听 `127.0.0.1` 的 **bridge WebSocket** 与常驻的 Claude Code 会话通信。
- 每次回复前**组装上下文**（见下「记忆系统」）。
- 回复后**解析标记**（见下「标记系统」）并执行副作用。
- 跑一个**主动性引擎**（每 5 分钟一拍）：主动找人、夜间内省、发「朋友圈」、记忆整合、情绪打点、运维告警等。

### Claude Code（cc）—— 主 AI 的「人」
主 AI 不是无状态的 API 调用，而是一个**常驻的 Claude Code 会话**（跑在 tmux 里），有自己的 `CLAUDE.md` 人设、MCP 工具、持久上下文。这让「他」有连续的性格和记忆感。hub 把用户消息 + 组装好的上下文经 bridge 喂给它，它回复里可带标记。

> 权衡：持久会话给「连续的人」的体感，但会话膨胀 / 压缩时会「失忆」——所以 hub 侧的记忆注入 + 夜间自动 `/compact` 是必要的兜底。

### 一次性 cc 进程
做梦、写日记、生成情绪打点、每日任务面板等**离线内容生成**，不走持久会话，而是 `spawn claude -p`（独立 `--session-id`，绝不污染主会话）。产物写进对应存储，前端/TG 直接读。

### 前端 PWA
React/Vite 单页应用，纯客户端；所有数据经 hub 的 HTTP API。多个「页」（聊天、记忆长河、潮汐心海/心跳、承诺、任务、账本、日记、小窝、朋友圈…）各自 fetch 对应端点。

## 一条消息的一生

1. 用户在**任意入口**发一句话 → 请求打到 hub。
2. hub 组装上下文：**人设**（带 prompt 缓存）＋ **检索出的相关记忆**＋ **最近记忆保底**＋ **最近对话回顾**＋ 任务/账本/健康/位置等**背景**。
3. 交给主 AI（持久 cc 会话，或备用 LLM）生成回复。
4. hub **解析回复里的标记**（任务编辑、承诺、账本、开网页…），执行副作用，并从正文里**剥掉标记**（用户看不到）。
5. 剥净的回复写进**共享聊天记录**，广播给所有在线入口（SSE/WS + TG 镜像）。
6. **记账**：这个入口欠的消息数够阈值了？起后台任务把这段对话**总结成记忆**，追加入库。

## 记忆系统（「不失忆」的关键）

三层，全部在网关侧做，所以任何入口天生共享：

### 短期 —— 交叉/统一的最近对话
所有入口写进**同一条聊天记录**，回复前注入最近 N 轮。因为是一条流，天然解决「话说到一半换了 App」的断裂——换到哪边都接得上，成本几乎为零。

### 中期 —— 自动摘要沉淀
给聊天记一本「账」，**当前消息数 − 已结账进度 ≥ 阈值** 就起后台任务，把这段对话总结成记忆、原子追加进库。

> ⚠ 两个踩过的坑：
> 1. **触发别用「总消息数取模」**：服务一重启计数器重新播种，播成奇数就永远撞不上阈值，自动记忆无声哑掉。用**账本差值制**（天然抗重启）。
> 2. **摘要要带人设语境**：裸把（尤其私密的）对话丢给无语境的模型做总结容易被拒，拒答文本一旦当记忆入库会污染检索。让摘要请求带上同款 persona，再加一层**拒答检测兜底**。

### 长期 —— 相关性检索 + 保底
每次回复前，用当前这句话检索最相关的若干条记忆注入；**无论命中与否，始终追加最近若干条记忆保底**，防止「聊着聊着忘了最近发生的事」。被带出的记忆会强化（heat +，`last_recalled` 更新）——用得上的越记越牢，配合定期热度衰减 = 「用进废退」。

> 检索起步用轻量的中文二元组(bigram)重叠打分（数百条时全表扫描即时）；到万级规模再上向量(如 bge-small-zh ONNX) + BM25 的 RRF 混合检索。

## 标记系统（让 AI 真的「动手」）

AI 不能直接调副作用，但可以在回复文本里**内嵌标记**；hub 解析→执行→从正文剥掉（用户看不到标记）。这套让一次纯文本回复就能同时「说话 + 做事」：

| 标记 | 作用 |
|---|---|
| `[加任务:名\|要求]` `[记完成:名]` … | 直接改今日任务面板 |
| `[承诺:内容]` | 记进「承诺」页 |
| `账本：+规矩/+欠/✓` | 写持久账本 |
| `[网页:url]` | 让 AI 打开网页（配浏览器工具） |
| `⟪拨号:理由⟫` `⟪挂断⟫`（语音扩展） | 应用内来电/挂断 |

解析是一条**管道**（`parseToy(parseQuest(parseLedger(...)))`），在所有回复路径（主聊/副 persona/群聊）统一走一遍。

## 主动性引擎

一个每 5 分钟的 `driveTick()`，各子模块自带门控（多数锁在 `DESIRE_DRIVEN` 开关后，关闸即回退成纯被动、零行为改变）：
- **主动找人 / 催睡**：长时间没影就找你、深夜还醒着就催睡（尊重「晚安」意图 → 安静到次日）。
- **夜间内省**：你睡着时让 AI 静默写日记/记梦/更新自我画像，绝不发消息打扰。
- **自动朋友圈**：白天自己发「朋友圈」进流里。
- **记忆整合做梦**：热度衰减 + 定期提炼 insight。
- **情绪打点**：定期写一条「此刻的情绪」(V/A 模型)，喂「潮汐心海/心跳」页。
- **会话轮换**：深夜你睡熟、cc 空闲时自动 `/compact` 一次，治会话膨胀。
- **运维告警**：磁盘/内存/崩溃循环 → 推 TG（限流去重）。

## 子系统一览

- **任务/账本（quest / ledger）**：Dom 视角的游戏化每日任务面板 + 持久规矩/欠账。AI 经标记增改删、替用户勾完成。
- **情绪 / 心跳（emotions）**：AI 定期打点的 V/A 情绪 + 独白，画象限图 + 时间线。
- **朋友圈 / 梦 / 日记 / 小窝**：各类 AI 自主生成的内容流。
- **运维（ops）**：`/status` 服务状态、`/logs` 日志、`/restart` `/shutdown` 远程管理（白名单 + 二次确认 + 审计）、报到打卡、自动告警。

## 部署

- **进程**：`systemd` 服务 → 一个 `supervise` 脚本 → 用 tmux 拉起 **hub 会话** + **cc 会话**（两个独立 tmux 会话）。supervise 检测 hub 不在监听就自动重启。
- **反向代理**：Caddy / nginx 前置 HTTPS（TG Mini App 与 PWA 必须 HTTPS）。
- **热更新技巧**：
  - `tmux kill-session -t hub` → supervise 约 30 秒内重建 hub，**保留 cc 会话**（最温柔的部署，不打断主 AI 的上下文）。
  - `systemctl restart` 不会重置 cc（tmux server 持久）。要彻底重开 cc 才 kill tmux server。
- **部署前务必**：`bun build hub.ts` 过一遍当语法闸；改完 scp 上去后 **grep 确认新代码真落地**再重启（别被 `scp | tail` 吞掉的失败退出码骗了）。

## 安全模型

- **安全词**：一条命令即全停（停主动、只安静陪着），压过一切。
- **运维白名单**：`/restart` `/shutdown` `/logs` 等管理动作只允许白名单 TG id（默认只认首个绑定的私聊），回调里二次校验，每次动作写审计日志。
- **bridge 只监听 `127.0.0.1`**；对外端点建议置于反代 + 鉴权之后。
- **隐私**：所有数据自托管、不出你的服务器；语音/视觉/TTS 若用云 API 需知悉「哪些内容会出境」（见 `.env.example` 注释）。
